Dynamic Test Engine for z/OS (Host Component)

El corazón de la capacidad de grabación y reproducción de TAZ. Se instala en z/OS y es una característica del z/OS Debugger.

  • z/OS 2.4 o superior.
  • z/OS Debugger v16.0.6 o v17.0.1.
  • Sistema de archivos UNIX de z/OS montado con el bit `SETUID` activo.
  • Acceso y permisos para actualizar PARMLIB, PROCLIB y definiciones de seguridad (RACF).
  1. Instalación SMP/E: Seguir el Program Directory correspondiente.
  2. Actualizaciones de PARMLIB:
    • `IFAPRDxx`: Habilitar producto `IBM Test Accelerator for Z` (ID 5900-BBG).
    • `LPALSTxx`: Añadir `EQAW.SEQALPA` a la LPA.
    • `PROGxx`: Autorizar con APF la librería `EQAW.SEQAMOD`.
    • `COMMNDxx`: (Para CICS/IMS) Añadir `S EQAPROF` para iniciar el Debug Profile Service.
  3. Actualizaciones de PROCLIB: Copiar y personalizar el PROC `EQAPPLAY` de `EQAW.SEQASAMP` a `SYS1.PROCLIB`.
  4. Definiciones de Seguridad (RACF): Activar clases (`FACILITY`, `STARTED`), definir la tarea `EQAPROF` con un ID protegido y otorgar permisos para actuar como servidor UNIX seguro y gestionar certificados.
  5. Configurar Debug Profile Service: Personalizar `eqaprof.env` (puerto, seguridad) y `dtcn.ports` (mapeo de regiones CICS).
  6. Integración con Subsistemas:
    • CICS: Actualizar CSD, parámetros de SIP (`DEBUGTOOL=NO`, `TCPIP=YES`) y JCL de la región.
    • IMS: Añadir `EQAW.SEQAMOD` al `STEPLIB` de las regiones de mensajes y `IMSBATCH`.

Early Development Testing for z/OS (Cliente)

Permite la creación, ejecución y depuración de pruebas unitarias y de integración tempranas en un IDE basado en Eclipse (IDz).

  1. Prerrequisitos: IBM Developer for z/OS (IDz) v16.0.6+ o v17.0.1 en Windows (x86-64).
  2. Instalación: Usar IBM Installation Manager o Eclipse P2 para añadir TAZ como una extensión a IDz.
  3. Configuración en Eclipse: Ir a `Window > Preferences > Early Development Testing for z/OS` para configurar HLQ para archivos de grabación y otras opciones.
  1. Crear Caso de Prueba: Clic derecho en un fuente COBOL local > `New > Unit Test`.
  2. Grabar Datos: En el Test Case Editor, usar el botón `Start recording data` para CICS, IMS o Batch. Ejecutar la transacción/job en el host.
  3. Generar y Construir: Usar el botón `Generate` para crear el programa de prueba y construirlo (ej. con DBB).
  4. Ejecutar y Depurar: Clic derecho en el archivo `.tc` > `Run As` o `Debug As`.
  5. Ver Resultados: Se muestran en la vista `Unit Test Runner Results` y pueden exportarse a JUnit o PDF.

Integration Test Builder for z/OS (Cliente)

Acelera la creación de pruebas de integración automatizadas y desplegables utilizando el framework Galasa en VS Code.

  1. Prerrequisitos: VS Code, Zowe Explorer, Zowe Explorer for CICS, IBM Semeru JDK 11/17, Maven.
  2. Instalación: Instalar la extensión `.vsix` desde VS Code. Instalar IBM Distribution for Galasa.
  3. Configuración en VS Code:
    • Usar el comando `> Create Galasa Project`.
    • Editar `.taz/itbz/itbz-config.yaml` para configurar ID de usuario, DBRM, dataset de grabación, etc.
    • Configurar `zowe.config.user.yaml` con las credenciales y el puerto del Debug Profile Service.
  1. Grabar Caso de Prueba: Usar los comandos `> IBM Integration Test Builder for z/OS: Start/Stop recording`.
  2. Interactuar con la Aplicación: Usar un emulador 3270 para ejecutar la transacción a grabar.
  3. Generar Prueba Galasa: Clic derecho en el archivo de reproducción (`.rec`) > `Generate Galasa integration test`.
  4. Personalizar y Ejecutar: Editar el archivo Java generado para añadir aserciones. Ejecutar la prueba desde la línea de comandos con `galasactl runs submit local`.

On-demand Dev and Test environment for z/OS

Permite el aprovisionamiento rápido y bajo demanda de instancias z/OS completas en entornos Linux, ideal para desarrollo y pruebas aisladas.

  1. Prerrequisitos: Máquina Linux (RHEL/Ubuntu) en x86-64 o s390x.
  2. Instalar Servidor de Licencias: Ejecutar el instalador, obtener un archivo de actualización desde IBM LKC y aplicarlo.
  3. Instalar "Tools Server": Ejecutar el instalador `odtz-install-1.0.0.tgz`.
  4. Configurar vía Web UI (`https://:9443`):
    • Conectar al servidor de almacenamiento (SFTP).
    • Añadir el servidor de licencias.
    • Definir usuarios, roles y entornos de origen/destino.
  1. Crear Componentes: Desde la UI del "Tools Server", extraer componentes (volúmenes, datasets) del entorno z/OS de origen.
  2. Crear Imágenes: Combinar los componentes en una "Imagen" desplegable. Se puede partir de una extracción personalizada o del "Stock Image" de IBM.
  3. Aprovisionar Instancia: Seleccionar una imagen y un entorno de destino para desplegar una nueva instancia z/OS virtualizada.
  4. Gestionar Instancias: Usar la `Instance Controller UI` o la línea de comandos del host Linux para iniciar, detener e interactuar con la instancia.

Detalles Críticos y Troubleshooting

Puntos Clave a Recordar

  • ➤
    Cambio de Nomenclatura `BZU*` a `EQA*`: Los casos de prueba creados con ZUnit (prefijo `BZU*`) DEBEN ser regenerados y reconstruidos para usar los nuevos artefactos `EQA*` proporcionados por z/OS Debugger. Esto es fundamental para la compatibilidad.
  • ➤
    Localización de `EQAPPLAY`: Este PROC es un ejemplo que se encuentra en `EQAW.SEQASAMP`. Debes copiarlo a tu PROCLIB del sistema y adaptarlo a las librerías y convenciones de tu entorno.
  • ➤
    Generación de Datos de Prueba (TDG): Esta potente función se inicia desde una prueba ya grabada para explorar rutas de código no cubiertas, no desde cero.

Solución de Problemas Comunes

  • ➤
    Error `Profile_Already_Exists_With_Same_Owner`: Significa que una grabación ya está en progreso en el Debug Profile Service. Debes detener la sesión activa antes de iniciar una nueva.
  • ➤
    Fallos en `galasactl` (Integration Test Builder): Verifica que `JAVA_HOME` apunte a una versión soportada de IBM Semeru JDK (11 o 17) y que el CLI de Galasa esté en el `PATH` de tu sistema.
  • ➤
    Consulta de Logs: Tu primer paso para diagnosticar cualquier problema. La documentación detalla la ubicación de los logs para cada componente. Por ejemplo, los logs del Tools Server están en `/liberty/usr/servers/server-instance/logs/`.
