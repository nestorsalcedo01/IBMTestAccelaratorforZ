Guía del Ecosistema de Pruebas Moderno

Implementando una estrategia de pruebas nativa para DevOps en z/OS con VS Code, ITB y Galasa.

1. El Ecosistema: Componentes Clave

Estación de Trabajo

  • VS Code: Editor de código principal.
  • Extensión ITB: Graba y genera pruebas.
  • Zowe Explorer: Puente con el mainframe.
  • JDK, Maven, Git: Herramientas de desarrollo.

Automatización

  • IBM Galasa: Framework para ejecutar las pruebas.
  • Orquestador CI/CD: Jenkins, Azure DevOps, etc.
  • WCA4Z: (Opcional) Acelerador de pipelines.

Host (Mainframe)

  • Dynamic Test Engine: Graba las interacciones de la aplicación (CICS, IMS, Db2).

2. Flujo de Implementación y Configuración

1

Configurar el Entorno Local

Instala todas las herramientas necesarias en tu estación de trabajo para conectar, grabar y generar pruebas.

✓ Instalar Software: VS Code, Git, JDK, Maven.

✓ Configurar Zowe Explorer: Instalar extensiones (Zowe, Zowe CICS) y crear perfil de conexión al Debug Profile Service (puerto 8143).

✓ Instalar Galasa: Añadir `galasactl` al PATH y configurar `settings.xml` de Maven.

✓ Instalar Extensión ITB: Instalar el fichero `.vsix` en VS Code.

2

Crear y Configurar un Proyecto Galasa

Genera la estructura de tu proyecto de pruebas y ajústala para apuntar a tu sistema y aplicación.

✓ Crear Proyecto: Usar el comando `> Create Galasa Project` en VS Code.

✓ Configurar Proyecto: Editar `.taz/itbz/itbz-config.yaml` para definir `userid`, `recordingDataset`, y `transactionid` a grabar.

3. El Ciclo de Vida de una Prueba Automatizada

1. Grabar

Usa los comandos de ITB en VS Code para iniciar y detener la grabación de una transacción en el host.

2. Generar

Haz clic derecho sobre el fichero `.rec` descargado para generar un script de prueba Java/Galasa.

3. Ejecutar

Usa `mvn` o `galasactl` en la terminal para ejecutar la prueba y verificar los resultados.

4. Versionar

Guarda el proyecto de prueba completo en Git para colaborar y para la automatización CI/CD.

4. Integración con DevOps y WCA4Z

Pipeline de CI/CD

  1. Disparador: Un `push` a Git activa el pipeline.
  2. Clonado: El orquestador (Jenkins, etc.) clona el repositorio con las pruebas de Galasa.
  3. Build y Ejecución: Se ejecuta `mvn verify` para compilar y lanzar las pruebas contra el host.
  4. Reporte: Los resultados en formato JUnit se publican en el dashboard del pipeline. Un fallo en la prueba detiene el build.

Relación con WCA4Z

WCA4Z (Wazi Code Automation for Z) es un conjunto de pipelines de referencia que acelera la adopción de DevOps en z/OS.

Las pruebas de Galasa creadas con ITB son un **componente ideal** para ser ejecutado dentro de un pipeline de WCA4Z, añadiendo una capa de validación de integración robusta al flujo de CI/CD existente.
