2.1 Configuración de Sistema (PARMLIB)

  • Habilitación del Producto: Añadir definición a IFAPRDxx para activar la licencia.
  • Definición de LPA: Añadir EQAW.SEQALPA a LPALSTxx para cargar los SVCs.
  • Autorización APF: Autorizar EQAW.SEQAMOD en PROGxx para CICS.

2.2 Configuración de Seguridad (RACF)

  • Definir Tarea Iniciada: Crear usuario y perfil para EQAPROF.* en la clase STARTED.
  • Conceder Privilegios: Dar permisos BPX.SERVER y BPX.SRV.** al usuario de la tarea.
  • Comunicación Cifrada: Crear certificados y llavero (keyring) para la comunicación segura del servicio.

Despliegue de herramientas cliente como dos sub-proyectos distintos: uno para desarrolladores tradicionales (EDT en IDz) y otro para testers/desarrolladores modernos (ITB en VS Code).

Flujo de Trabajo: EDT

  1. Crear Caso de Prueba (.tc) en IDz.
  2. Grabar datos de una transacción CICS/IMS.
  3. Editar datos de entrada y resultados esperados.
  4. Generar, construir (con DBB) y ejecutar la prueba.

Flujo de Trabajo: ITB

  1. Crear Proyecto Galasa en VS Code.
  2. Configurar itbz-config.yaml y perfil Zowe.
  3. Grabar caso de prueba para obtener un .rec.
  4. Generar y ejecutar la prueba Java basada en Galasa.

Lista de Verificación Inicial

  • ⚠Verificar Servicio Host: Comprobar que la tarea EQAPROF esté activa.
  • ⚠Verificar Conectividad: Usar telnet al puerto del servicio (ej. 8143) para confirmar acceso.
  • ⚠Revisar Logs del Servidor: Examinar logs en /var/debug en z/OS UNIX.
  • ⚠Confirmar Seguridad: Validar que todos los permisos RACF están aplicados y refrescados.

Aprovisionamiento de Entornos Virtuales

  • Servidor de Herramientas: Se instala en Linux y proporciona una UI web para la gestión de entornos.
  • Creación de Imágenes: El equipo de infraestructura crea "imágenes" de z/OS, ya sea personalizadas (extraídas de un z/OS real) o basadas en imágenes "stock" de IBM.
  • Auto-Aprovisionamiento: Los desarrolladores y testers pueden solicitar y desplegar sus propias instancias de z/OS a partir de las imágenes disponibles, reduciendo cuellos de botella.

Ambos caminos, EDT e ITB, son fundamentales para una estrategia de DevOps completa en z/OS, cada uno con un propósito específico.

Camino DevOps con EDT (Pruebas Unitarias)

  1. Fuente en Git: El desarrollador versiona el caso de prueba (.tc) y el COBOL de prueba generado.
  2. Build con DBB: El pipeline de CI invoca a IBM Dependency Based Build para compilar el caso de prueba automáticamente.
  3. Ejecución por JCL: El pipeline somete un JCL para ejecutar la prueba en el host.
  4. Transformación a JUnit: El resultado XML se convierte a formato JUnit usando las hojas de estilo XSLT incluidas.
  5. Publicación: El informe JUnit se publica en dashboards de CI/CD (Jenkins, SonarQube) para visibilidad.

Camino DevOps con ITB (Pruebas de Integración)

  1. Fuente en Git: El tester/desarrollador versiona el proyecto Galasa (código Java/Maven).
  2. Ejecución Nativa: El pipeline de CI invoca directamente a Galasa/Maven (ej. mvn clean install) para ejecutar las pruebas.
  3. Resultados Estándar: Los resultados se generan de forma nativa en formatos compatibles con herramientas de CI/CD.
  4. Uso Ideal: Perfecto para construir suites de pruebas de regresión automatizadas y robustas.

Gobernanza y Centro de Excelencia (CoE)

  • ✓Gestión de Código Fuente: Almacenar todos los artefactos de prueba (.tc, .itc, proyectos Galasa, configs) en un SCM moderno como Git.
  • ✓Convenciones de Nomenclatura: Establecer estándares para casos de prueba e imágenes para mantener la trazabilidad.
  • ✓Centro de Excelencia (CoE): Formar un grupo de expertos para guiar, definir mejores prácticas y acelerar la adopción en la organización.
